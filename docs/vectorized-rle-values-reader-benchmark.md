# `VectorizedRleValuesReader` Benchmark — Design and Baseline

Companion to `docs/vectorized-rle-values-reader-performance-analysis.md`. This document covers the low-level benchmark that isolates the RLE decoder so per-optimization gains (P0, P1, P3, P4) can be quantified without IO or decompression noise.

## 1. Why a dedicated low-level benchmark

End-to-end Parquet reads bury RLE-decoder costs under schema conversion, decompression, and IO. A 30% gain in the PACKED path can show up as 3-5% at the query level. To quantify each optimization cleanly we need a direct-call harness that:

- Calls the decoder entry points used in production (`readBooleans`, `readIntegers`, `readBatch`).
- Feeds hand-crafted RLE-encoded byte streams whose shape we control (PACKED-heavy, specific null density, specific bit width).
- Keeps allocation, setup, and JIT warmup out of the measured region so numbers reflect decode work only.

The benchmark extends `BenchmarkBase` (not `SqlBasedBenchmark`), mirroring the pattern of `ColumnarBatchBenchmark`.

## 3. Benchmark groups

| Group | Entry point | Varied dimension | Primary target |
|-------|-------------|------------------|----------------|
| A | `readBooleans(total, vec, 0)` | `trueRatio ∈ {0.1, 0.5, 0.9}` | **P1** (byte-at-a-time writes in PACKED boolean decode) |
| B | `readIntegers(total, vec, 0)` | `bitWidth ∈ {4, 8, 12, 20}` | **P3** (currentBuffer growth), **P4** (`ByteBuffer.slice` per 8-value group) |
| C | `readBatch(state, values, defLevels, constantReader, intUpdater)` | `nullRatio ∈ {0.0, 0.1, 0.3, 0.5, 0.9}` × `{random, clustered}` | **P0** (per-element virtual dispatch in PACKED null/non-null branch), isolated |
| D | same, with plain INT32 value reader | same | **P0**, realistic workload |

### Cold vs reused

Groups A and B each have two cases per dimension:

- **Cold** — a fresh `VectorizedRleValuesReader` is constructed inside the measured region. Exercises P3's cold buffer allocation path.
- **Reused** — reader is built and warmed once outside the closure (one full decode discarded). The measured region does only `initFromPage` + the read.

Groups C and D are single-variant per `(nullRatio, clustering)` pair because the 256-batch inner loop naturally amortizes cold setup; a separate cold case would add complexity for marginal insight.

## 4. What the benchmark deliberately excludes from the timed region

| Excluded | Why |
|----------|-----|
| Column vector allocation (4 MB `int[]`) | Not part of any optimization being measured. Allocated once per group, reused across cases. |
| Byte-buffer encoding via `RunLengthBitPackingHybridEncoder` | One-time setup per case. |
| Plain-reader byte buffer | `plainIntFactory` pre-allocates the 4 MB byte array; the `VectorizedPlainValuesReader` is re-initialised per iteration over the same buffer, avoiding per-iter large allocations. |
| First-read warmup (reused variants) | One full decode happens outside the closure so the reader's internal `currentBuffer` is sized and the JIT has seen the path. |

Dilution analysis (for 1M-row decodes expected at 5-50 ms per iter):
- Pre-restructure, with inline allocation: ~2-40% noise.
- Post-restructure: well under 1% (`initFromPage` and factory calls only).

## 5. Why a reflection bridge

`ParquetReadState` and its `resetForNewBatch` / `resetForNewPage` methods are package-private. When the benchmark runs under `spark-submit --jars`, the sql-core test jar is loaded by `MutableURLClassLoader` while main sql-core lives on the app classloader — distinct runtime packages, so package-private access fails at link time.

Three realistic ways around this:

| Option | Prod change | Reflection |
|--------|-------------|------------|
| A: widen `ParquetReadState` to public | Yes (4 modifier edits) | None |
| B: helper class in `main/` | Yes (new class) | None |
| B′: helper class in `test/` | **None** | Inside test helper only |
| C: reflection inline in benchmark | None | In benchmark |

We chose **B′**: `ParquetReadStateTestAccess` lives in `sql/core/src/test/java/...`, uses `setAccessible(true)` internally, and exposes clean public methods (`newState`, `resetForNewBatch`, `resetForNewPage`, `readBatch`). Reflection is confined to the helper; the benchmark holds `state` as `AnyRef` and sees no reflection noise in its own bytecode. Reflection cost is paid twice per batch (`resetForNewBatch` + `readBatch`) over 256 batches per iter — microseconds total, amortized to nothing across 1M decoded rows.

## 6. Value-reader factory pattern

`readBatch` needs an inner `VectorizedValuesReader` for actual value decoding. Two variants:

- `ConstantIntValuesReader` — returns zeros, no decoding work. Used in Group C to isolate the RLE decoder's per-element overhead. All non-int methods throw, keeping the class focused.
- `VectorizedPlainValuesReader` (Spark's production reader) — decodes plain-encoded INT32 bytes. Used in Group D for realistic numbers.

Both are wrapped in a `ValueReaderFactory: () => VectorizedValuesReader` closure. The plain factory pre-allocates a 4 MB byte buffer once per case and reinitialises the reader each iteration — no large allocation inside the timed region.

## 7. Baseline results

Run the benchmark via the `Run benchmarks` GHA workflow (class =
`*VectorizedRleValuesReader*`) to generate the three per-JDK result files listed in
§2. Expected result-file names:

- `sql/core/benchmarks/VectorizedRleValuesReaderBenchmark-results.txt` (JDK 17)
- `sql/core/benchmarks/VectorizedRleValuesReaderBenchmark-jdk21-results.txt`
- `sql/core/benchmarks/VectorizedRleValuesReaderBenchmark-jdk25-results.txt`

Once generated, the key cross-JDK signals to look for are:

- **Group A (P1 target):** cold ≈ reused ns/row across all JDKs; per-element `putByte` dominates.
- **Group B (P3, P4):** ns/row scales with bit width; cold = reused confirms P3 is latency-invisible.
- **Group C (P0 isolated):** random/clustered gap at mid-null densities (≈2× at `nullRatio=0.5`) is the algorithmic signal — stable across JDKs because it's not JIT-sensitive.
- **Group D (P0 realistic):** Group C shape shifted up by plain-reader decoding cost.

## 8. Signals and what to watch when optimizations land

| Optimization | Expected result-file signal |
|--------------|-----------------------------|
| **P0** null-run batching | Random cases in Groups C and D drop toward clustered numbers. Biggest wins at `nullRatio=0.3` and `0.5`. Clustered cases improve modestly. |
| **P1** boolean batch writes | All Group A cases drop; mid trueRatios likely gain most (fewer branch-prediction hits). |
| **P3** buffer-growth strategy | Barely moves the needle in warm cases; check `cold` rows in Group B — marginal improvement expected. |
| **P4** `ByteBuffer.slice` removal | All Group B cases shift down uniformly. Higher bitWidths (12, 20) gain slightly more since they have more PACKED groups per page. |
